# 订阅重复购买与管理员重复分配实施方案

## 1. 目标

在不改变现有默认行为的前提下，为每个订阅套餐配置重复获得权益时的处理方式，并同时覆盖以下来源：

- 外部支付成功后的订阅发放。
- 用户余额购买。
- 管理员给指定用户手动分配订阅。

实现需保持 SQLite、MySQL、PostgreSQL 兼容，不改动各支付渠道回调协议，不新增订阅路由，并尽量把自定义逻辑放在新文件中，降低后续 `main` 合并成本。

## 2. 当前行为

所有购买和管理员分配最终都调用 `model.CreateUserSubscriptionFromPlanTx`。当前每次调用都会创建一条新的 `UserSubscription`：

- 开始时间为当前时间。
- 结束时间按套餐时长从当前时间计算。
- 总额度取套餐额度，已用额度为零。
- 同一用户、同一套餐的多条订阅相互独立。
- 消耗时按最早到期顺序选择一条能够独立承担整笔请求的订阅，不跨记录拆分扣除。
- `MaxPurchasePerUser` 按历史订阅记录条数统计，管理员分配也受该限制。

这意味着当前重复购买不会自动延长时间或合并额度，管理员也无法在达到购买上限后补发订阅。

## 3. 重复处理模式

在 `SubscriptionPlan` 增加 `repeat_purchase_mode`，默认值为 `independent`。支持以下固定模式：

| 模式 | 时间处理 | 额度处理 |
| --- | --- | --- |
| `independent` | 创建独立订阅 | 创建独立额度 |
| `extend_time` | 从现有结束时间继续增加一个套餐周期 | 保持总额度和已用额度 |
| `add_quota` | 保持现有开始、结束时间 | 增加套餐额度，保持已用额度 |
| `extend_time_add_quota` | 延长一个套餐周期 | 增加套餐额度，保持已用额度 |
| `max_validity` | 采用现有结束时间与本次结束时间中的较晚值 | 保持总额度和已用额度 |
| `max_validity_add_quota` | 采用现有结束时间与本次结束时间中的较晚值 | 增加套餐额度，保持已用额度 |
| `extend_time_reset_quota` | 从现有结束时间继续增加一个套餐周期 | 使用本次套餐额度并清零已用额度 |
| `replace` | 从当前时间重新计算开始、结束时间 | 使用套餐额度覆盖并清零已用额度 |

规则补充：

- 仅合并状态为 `active`、尚未到期的同套餐订阅。
- 没有有效目标时，所有模式都创建新订阅。
- 存在多条有效目标时，选择结束时间最晚、ID 最大的一条；其他历史记录不删除、不失效。
- `AmountTotal == 0` 代表无限额度，任一参与叠加的额度为无限时，结果仍为无限。
- 有周期重置的套餐叠加额度后，后续周期会恢复到叠加后的总额度；一次性临时额度不属于本次范围。
- 所有额度加法必须检查整数溢出。

## 4. 用户购买与管理员分配

底层应用订阅时使用明确的上下文参数：

```go
type SubscriptionApplyOptions struct {
    Source               string
    ApplyMode            string
    EnforcePurchaseLimit bool
}
```

不同来源的行为如下：

| 来源 | 应用模式 | 是否执行购买上限 |
| --- | --- | --- |
| 外部支付 | 套餐默认模式 | 是 |
| 余额购买 | 套餐默认模式 | 是 |
| 管理员分配 | 默认跟随套餐，可在每次操作时覆盖 | 否 |

管理员可选 `apply_mode`：

- `plan_default`
- `independent`
- `extend_time`
- `add_quota`
- `extend_time_add_quota`
- `max_validity`
- `max_validity_add_quota`
- `extend_time_reset_quota`
- `replace`

管理员分配不因达到 `MaxPurchasePerUser` 被拒绝，但仍计入累计获得次数，因此后续用户自助购买仍受累计次数影响。这与现有“管理员分配也占用一次记录”的语义兼容，同时允许管理员执行补发。

## 5. 累计次数

在 `UserSubscription` 增加：

```go
AllocationCount int64 `json:"allocation_count"`
```

- 新建记录时为 1。
- 每次合并成功后加 1。
- 历史记录迁移时小于 1 的值统一回填为 1。
- 购买上限按同用户、同套餐所有记录的 `AllocationCount` 之和统计。
- 管理端和用户端显示的累计次数同样按该字段求和。

这样即使五次购买被合并为一条订阅，购买上限仍按五次计算。

## 6. 事务与并发

共享应用函数在同一事务中按以下顺序执行：

1. 锁定用户行，串行化同一用户的购买和管理员分配。
2. 统计累计获得次数并按来源决定是否执行购买上限。
3. 计算本次新权益快照。
4. 查询并锁定同套餐合并目标。
5. 按模式创建或更新订阅，并更新 `AllocationCount`。
6. 提交事务后更新用户分组缓存并记录日志。

外部支付订单继续由现有订单锁保证幂等；Epay 的通知和返回同时到达时不会重复叠加。

## 7. 分组、重置和来源快照

- 新建记录继续沿用现有升级分组、降级分组和钱包溢出快照逻辑。
- 合并时保留最早记录的 `PrevUserGroup`，避免重复购买后丢失原始回退分组。
- 套餐本次配置了升级或降级分组时更新对应快照。
- `extend_time` 在原订阅重置计划已因旧结束时间停止时，重新计算下一次重置时间。
- `add_quota` 不改变现有重置时间。
- `max_validity` 系列仅在结束时间实际变长时恢复已停止的重置计划。
- `extend_time_reset_quota` 保持正常运行的重置周期；原计划已停止时按延长后的结束时间恢复。
- `replace` 使用本次新权益的开始时间、结束时间和重置时间。
- 非覆盖合并保留原记录的 `Source`；管理员操作通过管理审计记录，不依赖 `Source` 表达完整历史。

## 8. 管理端交互

Default 和 Classic 的用户订阅管理弹窗增加“重复分配处理”选择框，默认“跟随套餐默认策略”。

管理员提交后返回：

- 创建还是合并。
- 实际应用模式。
- 目标订阅 ID。
- 更新后的有效期、额度和累计次数。

`replace` 会清零已用额度并重算有效期，提交前必须二次确认。订阅列表增加累计分配次数展示。

管理操作复用现有 `RecordLogWithAdminInfo` 和 `recordManageAuditFor`，记录管理员、目标用户、套餐、实际模式、目标订阅以及操作前后的时间和额度。

## 9. API 兼容

现有路由保持不变。管理员创建订阅请求扩展为：

```json
{
  "plan_id": 1,
  "apply_mode": "plan_default"
}
```

`apply_mode` 缺省时按 `plan_default` 处理。由于现有套餐迁移后的默认模式为 `independent`，旧客户端行为不变。

套餐创建、更新和查询继续使用现有 API，仅在套餐 JSON 中增加 `repeat_purchase_mode`。

## 10. 代码组织与上游合并

新增文件：

- `model/subscription_repeat_purchase.go`
- `model/subscription_repeat_purchase_test.go`

现有核心文件只保留必要接入点：

- `model/subscription.go`：模型字段、共享创建入口和调用参数。
- `model/main.go`：SQLite 套餐补列和历史累计次数回填。
- `controller/subscription.go`：套餐字段校验、管理员模式参数和审计。

不修改各支付渠道请求与回调文件，不增加路由，不建立单独策略表，不引入规则 DSL。Default 和 Classic 只修改现有套餐表单、用户订阅弹窗、类型和累计次数计算。

## 11. 测试计划

### 后端模型测试

- 默认独立模式连续应用产生多条订阅。
- 八种模式在无目标、有目标、过期目标和多目标时结果正确。
- 仅延长时间不改变额度；仅叠加额度不改变有效期。
- 覆盖模式清零已用额度并重算重置周期。
- 无限额度和额度溢出边界正确。
- 合并后累计次数和购买上限正确。
- 管理员可绕过上限，但仍增加累计次数。
- 外部支付重复回调保持幂等。
- 分组快照和重置计划在合并后保持正确。
- 合并后的单条额度能够承担叠加前无法由单条记录承担的请求。

### 前端验证

- 套餐创建、编辑可保存并重新加载默认模式。
- 管理员可以跟随套餐或覆盖模式。
- `replace` 取消确认时不发送请求。
- 操作完成后列表显示正确有效期、额度和累计次数。
- 用户购买上限按累计次数而不是记录条数计算。

### 回归与本地运行

- `go test ./...`
- Default：typecheck、涉及文件 lint、生产构建。
- Classic：生产构建。
- `git diff --check`
- 使用全新本地 SQLite 数据库启动服务，确认健康检查、数据库迁移和前端静态资源正常。
- 通过模型集成测试覆盖完整业务矩阵；本地服务启动用于验证实际启动、迁移和路由可用性。

## 12. 不在本次范围

- 一次性额度在下个重置周期自动失效。
- 跨不同套餐合并、升级补差价或额度折算。
- 管理员指定某一条订阅作为合并目标。
- 自动破坏性合并历史多条订阅。
- 管理员赠送不计入购买上限。

这些能力需要独立权益流水或更细的业务定义，当前需求不提前实现。

八种重复处理模式、有效/过期/多目标选择、累计次数、购买上限、管理员绕过上限与单次覆盖、支付回调幂等、余额购买、额度无限与溢出、重置计划、用户分组快照及合并后大额单次扣费，均由确定性的模型集成测试覆盖。临时数据库和日志仅用于本地验证，不写入仓库。
